# 混合方案：字段定义纯代码 + 表名LLM生成

## 问题分析

用户指出当前纯代码逻辑无法灵活适配不断变化的开发规范。

## 解决方案

采用混合方案：
1. **字段定义**：继续使用纯代码生成（速度快、语法正确）
2. **表名生成**：交给LLM，让它根据开发规范生成符合要求的表名

## 实施计划

### 步骤1：创建表名生成提示词模板

在 `prompts.py` 中添加表名生成专用的提示词模板，包含：
- 开发规范（从 standards.json 读取）
- 数据库类型要求
- 建表示例（从 db_examples.py 读取）
- 当前表的中文名称和字段信息

### 步骤2：创建表名生成方法

在 `FieldProcessor` 中添加 `generate_table_name` 方法：
- 构建表名生成的提示词
- 调用LLM生成英文表名
- 解析LLM返回的表名

### 步骤3：修改DDL生成流程

修改 `generate_ddl_for_table` 方法：
- 先调用LLM生成英文表名
- 再用纯代码生成字段定义和DDL结构

### 步骤4：更新提示词配置

确保开发规范可以被动态加载并传递给LLM。

## 方案流程图

```
Excel数据
    ↓
提取中文表名和字段
    ↓
┌─────────────────────────────────────┐
│ 调用LLM生成英文表名                 │
│ 输入：                              │
│ - 中文表名                          │
│ - 开发规范（表命名规则）             │
│ - 数据库类型                        │
│ - 建表示例                          │
│ 输出：英文表名（如 dwd_fin_order）  │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 纯代码生成字段定义                   │
│ 使用历史词根匹配结果                 │
│ 生成字段名和数据类型                │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 组装完整DDL                        │
│ 根据数据库类型选择正确格式           │
└─────────────────────────────────────┘
```

## 提示词模板设计

```python
TABLE_NAME_PROMPT = """
【任务】根据以下信息生成符合规范的英文表名

【中文表名】{chinese_table_name}

【开发规范】
{standards_content}

【数据库类型】{db_type}

【建表示例】
{db_example}

【输出格式】
请只输出英文表名，不要包含任何解释或额外内容。

示例输出：
dwd_fin_order
"""
```

## 优势分析

| 维度 | 纯代码方案 | 全LLM方案 | 混合方案（推荐） |
|------|----------|----------|----------------|
| 速度 | 快 | 慢 | 较快 |
| 规范遵从性 | 差（硬编码） | 好 | 好 |
| 灵活性 | 差 | 好 | 好 |
| 可维护性 | 差（需改代码） | 好 | 好 |

## 风险处理

1. **LLM调用失败**：使用fallback表名生成逻辑
2. **表名不符合规范**：最终校验阶段进行修正
3. **性能问题**：缓存表名生成结果

## 预期效果

修改后：
- 开发规范修改后只需更新配置文件，无需改代码
- LLM会根据最新规范生成表名
- 字段定义仍然快速生成
- 最终DDL既符合规范又保证语法正确
