# 大模型翻译器架构优化计划

## 一、当前架构分析

### 1.1 现有流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 当前架构：分词 → 匹配 → 翻译 → 拼接                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Step 1: jieba分词                                                       │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ "订单状态" → ["订单", "状态"]                                      │   │
│  │ "是否删除" → ["是否", "删除"]                                      │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                           ↓                                             │
│  Step 2: 历史词根匹配                                                     │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ "订单" → 命中历史词根 → "order"                                   │   │
│  │ "状态" → 命中历史词根 → "status"                                  │   │
│  │ "是否" → 未命中 → 需要LLM翻译                                     │   │
│  │ "删除" → 命中历史词根 → "delete"                                  │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                           ↓                                             │
│  Step 3: LLM翻译未匹配词根                                                │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ LLM输入: "是否"                                                   │   │
│  │ LLM输出: "is" (或错误输出"flag")                                  │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                           ↓                                             │
│  Step 4: 机械拼接                                                        │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ "订单" + "状态" → "order_status"                                  │   │
│  │ "是否" + "删除" → "is_delete"                                     │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.2 架构限制分析

| 限制点 | 问题描述 | 影响 |
|--------|----------|------|
| **分词与翻译分离** | jieba分词后直接翻译词根，LLM看不到完整字段语义 | 翻译可能偏离上下文 |
| **LLM只做翻译** | LLM只翻译单个词根，不参与字段名设计 | 无法利用LLM的语义理解能力 |
| **机械拼接** | 简单的下划线拼接，不考虑语义连贯性 | 可能产生不自然的字段名 |
| **上下文丢失** | 分词后丢失原始字段的完整语义 | "商品名称"分词后，LLM不知道这是"商品"的"名称" |
| **规范利用不足** | 开发规范只传给LLM做翻译参考 | 没有让LLM按规范设计字段名 |
| **批量翻译无关联** | 多个词根分批翻译，彼此独立 | 无法保证一致性 |

## 二、优化方案

### 2.1 方案A：LLM参与字段名设计（推荐）

**核心思路**：让LLM直接设计字段名，而不是只翻译词根

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 优化架构：分词 → 匹配 → LLM设计字段名                                      │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Step 1-2: 同上（分词 + 历史词根匹配）                                     │
│                           ↓                                             │
│  Step 3: LLM设计字段名（新增）                                            │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 输入:                                                            │   │
│  │   - 完整中文字段名: "订单状态", "是否删除", "商品名称"              │   │
│  │   - 分词结果: ["订单", "状态"], ["是否", "删除"], ...              │   │
│  │   - 已匹配词根: {"订单": "order", "状态": "status", ...}          │   │
│  │   - 未匹配词根: ["是否", ...]                                     │   │
│  │   - 开发规范: 完整规范内容                                         │   │
│  │                                                                  │   │
│  │ 输出:                                                            │   │
│  │   - "订单状态" → "order_status"                                  │   │
│  │   - "是否删除" → "is_delete"                                     │   │
│  │   - "商品名称" → "product_name"                                  │   │
│  │                                                                  │   │
│  │ LLM可以:                                                         │   │
│  │   1. 理解完整字段语义                                             │   │
│  │   2. 按开发规范设计                                               │   │
│  │   3. 保证跨字段一致性                                             │   │
│  │   4. 处理特殊情况（如"是否"→"is"）                                │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**：
- LLM能看到完整上下文，理解字段语义
- 能按开发规范设计字段名
- 能处理特殊情况（如"是否"→"is"）
- 能保证跨字段一致性

**缺点**：
- LLM调用次数可能增加
- 需要更复杂的提示词设计

### 2.2 方案B：增强词根翻译上下文

**核心思路**：在词根翻译时提供更多上下文

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 优化架构：分词 → 匹配 → 增强上下文翻译 → 拼接                               │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Step 3: LLM翻译（增强上下文）                                            │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 输入:                                                            │   │
│  │   - 待翻译词根: "是否"                                           │   │
│  │   - 出现上下文: ["是否删除", "是否有效", "是否启用"]              │   │
│  │   - 开发规范: 基础词根速查表                                      │   │
│  │   - 相关已翻译词根: {"删除": "delete", "有效": "valid"}           │   │
│  │                                                                  │   │
│  │ 输出:                                                            │   │
│  │   - "是否" → "is" (因为上下文都是布尔字段)                        │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**：
- 改动较小，只需增强提示词
- LLM能看到词根的使用场景

**缺点**：
- 仍然是词根级翻译，不是字段级设计
- 拼接仍是机械的

### 2.3 方案C：混合模式（推荐）

**核心思路**：历史词根匹配 + LLM字段设计 + 智能拼接

```
┌─────────────────────────────────────────────────────────────────────────┐
│ 混合架构：分层处理                                                        │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│  Layer 1: 完全匹配（无需LLM）                                             │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 所有词根都命中历史词根 → 直接拼接                                  │   │
│  │ "订单状态" → ["订单", "状态"] → ["order", "status"] → order_status│   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                           ↓                                             │
│  Layer 2: 部分匹配（LLM补充）                                             │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 部分词根未命中 → LLM翻译未匹配词根 → 拼接                          │   │
│  │ "是否删除" → ["是否"(未匹配), "删除"(匹配)]                       │   │
│  │           → LLM翻译"是否"→"is" + "delete"                        │   │
│  │           → is_delete                                           │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                           ↓                                             │
│  Layer 3: 完全未匹配（LLM设计 + 词根分解）                                 │
│  ┌─────────────────────────────────────────────────────────────────┐   │
│  │ 所有词根都未命中 → LLM设计字段名 + 返回词根分解                     │   │
│  │                                                                  │   │
│  │ 输入: "特殊标识" - 分词: ["特殊", "标识"]                          │   │
│  │                                                                  │   │
│  │ LLM输出格式:                                                     │   │
│  │   中文字段名:英文字段名|中文词根1:英文词根1,中文词根2:英文词根2      │   │
│  │                                                                  │   │
│  │ 示例输出:                                                        │   │
│  │   特殊标识:special_id|特殊:special,标识:id                        │   │
│  │                                                                  │   │
│  │ 结果:                                                            │   │
│  │   - 字段名: special_id                                          │   │
│  │   - 新词根: {"特殊": "special", "标识": "id"}                   │   │
│  └─────────────────────────────────────────────────────────────────┘   │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

**优点**：
- 充分利用历史词根，减少LLM调用
- LLM只在必要时参与
- 分层处理，效率高
- **词根分解保证词根库正确保存**：新词根以分开的形式保存，可复用

### 2.4 Layer 3 详细设计

**提示词设计**：
```python
prompt = f"""请为以下中文字段设计英文字段名，并同时返回词根分解：

【完全未匹配字段列表】
这些字段的所有词根都未命中历史词根，需要重新设计：

1. "特殊标识" - 分词: ["特殊", "标识"]
2. "业务编码" - 分词: ["业务", "编码"]
3. "备用字段" - 分词: ["备用", "字段"]

【开发规范】
{standards}

【设计要求】
1. 按开发规范设计字段名
2. 字段名长度 ≤ 60字符
3. 使用下划线分隔，全部小写
4. 词根分解必须与分词结果一一对应

【输出格式】
每行一个：中文字段名:英文字段名|中文词根1:英文词根1,中文词根2:英文词根2,...

【输出示例】
特殊标识:special_id|特殊:special,标识:id
业务编码:biz_code|业务:biz,编码:code
备用字段:backup_field|备用:backup,字段:field
"""
```

**解析逻辑**：
```python
def parse_llm_layer3_output(output_line):
    """
    解析Layer 3的LLM输出
    输入: "特殊标识:special_id|特殊:special,标识:id"
    输出: {
        "field_name": "special_id",
        "roots": {"特殊": "special", "标识": "id"}
    }
    """
    # 分割字段名和词根分解
    parts = output_line.split('|')
    field_part = parts[0]  # "特殊标识:special_id"
    root_part = parts[1] if len(parts) > 1 else ""  # "特殊:special,标识:id"
    
    # 解析字段名
    chinese_field, english_field = field_part.split(':')
    
    # 解析词根分解
    roots = {}
    if root_part:
        for root_pair in root_part.split(','):
            cn_root, en_root = root_pair.split(':')
            roots[cn_root.strip()] = en_root.strip()
    
    return {
        "chinese_field": chinese_field,
        "field_name": english_field,
        "roots": roots
    }
```

**词根库保存**：

### 2.5 语义分组优化

**当前问题**：
```python
# 当前分批逻辑：简单按顺序每50个一批
for root in unmatched_roots:
    current_batch.append(root)
    if len(current_batch) >= MAX_FIELDS_PER_GROUP:
        batches.append(current_batch)
        current_batch = []
```

**问题**：
- 语义相关的词根（如"数量"、"数字"、"金额"）分散在不同批次
- 同一批次的词根之间没有语义关联
- LLM翻译时无法参考同批次其他相关词根

**优化方案：语义分组 + 排序**

```python
def group_roots_by_semantic(roots):
    """
    按语义分组词根
    语义相近的词根分在同一组，保证翻译一致性
    """
    # 语义分组定义
    semantic_groups = {
        "数量类": ["数量", "数字", "个数", "笔数", "次数", "频次", "人数"],
        "金额类": ["金额", "余额", "总额", "收入", "支出", "成本", "费用", "利润"],
        "标识类": ["标识", "编号", "编码", "ID", "号码"],
        "状态类": ["状态", "情况", "阶段"],
        "类型类": ["类型", "种类", "类别"],
        "名称类": ["名称", "姓名", "标题", "品名"],
        "日期类": ["日期", "年月日", "日", "月", "年"],
        "时间类": ["时间", "时刻", "时分秒"],
        "是否类": ["是否", "有无", "真假"],
        "描述类": ["描述", "说明", "备注", "摘要"],
    }
    
    # 按语义分组
    grouped = {name: [] for name in semantic_groups}
    ungrouped = []
    
    for root in roots:
        found = False
        for group_name, keywords in semantic_groups.items():
            if any(k in root for k in keywords):
                grouped[group_name].append(root)
                found = True
                break
        if not found:
            ungrouped.append(root)
    
    # 合并分组
    result = []
    for group_roots in grouped.values():
        if group_roots:
            result.extend(sorted(group_roots))
    
    # 未分组的词根放在最后
    result.extend(sorted(ungrouped))
    
    return result


def create_batches_with_semantic_order(roots, batch_size=50):
    """
    按语义分组创建批次
    同一语义的词根尽量在同一批次
    """
    # 1. 按语义分组并排序
    sorted_roots = group_roots_by_semantic(roots)
    
    # 2. 分批：同一语义的词根优先放在一起
    batches = []
    current_batch = []
    current_semantic = None
    
    for root in sorted_roots:
        # 检测当前词根的语义类别
        root_semantic = detect_semantic(root)
        
        # 如果当前批次满了，或者语义发生了变化（且当前批次已有内容）
        if len(current_batch) >= batch_size or (current_semantic and root_semantic != current_semantic and len(current_batch) >= batch_size // 2):
            batches.append(current_batch)
            current_batch = []
            current_semantic = None
        
        current_batch.append(root)
        if root_semantic:
            current_semantic = root_semantic
    
    if current_batch:
        batches.append(current_batch)
    
    return batches
```

**优化效果**：

| 词根列表 | 优化前批次 | 优化后批次 |
|----------|-----------|-----------|
| 金额,数量,ID,状态,数字,收入 | 混合: [金额,数量,ID,状态,数字,收入] | 分组: [金额,收入],[数量,数字],[ID],[状态] |
| 是否,名称,是否,类型,数量,是否 | 混合: [是否,名称,是否,类型,数量,是否] | 分组: [是否,是否,是否],[数量],[类型],[名称] |

**优势**：
1. LLM翻译时可以参考同批次的同语义词根
2. 保证翻译一致性（如"数量"和"数字"都翻译为"num"）
3. 减少同一语义用不同词根翻译的问题

### 3.2 上下文增强

**新增输入信息**：
- 完整中文字段名（不只是分词后的词根）
- 词根出现的所有上下文
- 相关已翻译词根
- 表名和表的其他字段（用于一致性）

---

### 3.3 批量处理优化

**当前**：分批翻译，每批50个词根，彼此独立

**优化**：
- 同一张表的字段放在同一批
- 批次间传递已翻译结果，保证一致性
- 特殊词根（如"是否"）单独处理

---

### 3.4 后处理优化

**当前**：机械拼接 `"_".join(english_parts)`

**优化**：
- 检查是否符合开发规范
- 检查是否与同表其他字段一致
- 特殊处理（如"is_"开头的字段）

## 四、实施步骤

### Phase 1: 提示词优化（低风险）
1. 修改 `_translate_roots_batch` 方法
2. 增强提示词，添加上下文信息
3. 添加完整字段名作为输入

### Phase 2: 分层处理（中风险）
1. 实现"完全匹配/部分匹配/完全未匹配"分层
2. 不同层采用不同处理策略
3. 减少不必要的LLM调用

### Phase 3: LLM字段设计（高风险）
1. 新增 `design_field_names_via_llm` 方法
2. 让LLM直接设计字段名
3. 保留词根翻译作为备选

## 五、风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM调用次数增加 | 成本上升、速度下降 | 分层处理，减少调用 |
| 提示词过长 | LLM可能忽略部分内容 | 精简提示词，突出重点 |
| LLM输出不稳定 | 可能产生不一致结果 | 增加约束，后处理校验 |
| 兼容性问题 | 现有功能可能受影响 | 保留旧逻辑作为备选 |

## 六、预期效果

| 指标 | 当前 | 优化后 |
|------|------|--------|
| "是否"翻译准确率 | ~70%（可能翻译为flag） | ~99%（强制翻译为is） |
| 字段名语义准确性 | 中等（机械拼接） | 高（LLM理解语义） |
| 跨字段一致性 | 低（独立翻译） | 高（批量处理+上下文） |
| LLM调用次数 | N个未匹配词根 | 分层后减少30-50% |
